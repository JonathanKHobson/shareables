The short version

You shared the Local Intelligence Packet - a vault-agnostic system for turning selected work into curated HTML intelligence packets. I turned it into a local MCP pipeline called Shareables MCP. It's now how I ship things. And this page is proof it works.

What you gave me

Cuts

A config that selects content, sets the voice, and tells the system who it's for and how to render it.

Frames

The rendered HTML output of a cut. Self-contained, portable, openable in any browser.

Lenses

Rendering patterns - Reel, Explainer, Pitch, Product Website, First Look, and more. Structure for every kind of story.

Windows

Multiple frames bundled into one URL with tabbed navigation. A whole briefing in one file.

Person profiles

Voice and audience preferences attached to a collaborator. Render differently for different readers.

Four design principles

Make harm legible. Provide an exit. Design for sufficiency. Scale out, not up. These run through everything.

What clicked for me

When I first opened the packet, I thought: smart folder, neat scripts. Then I read the design principles. Then the cuts spec. Then I realized it wasn't a folder - it was a repeatable discipline for turning selected work into curated artifacts and sharing them with specific people.

The part that landed hardest: it's not about having a perfect system. It's about having a portable one. Content stays as markdown. The pipeline adapts to your folder. You can leave at any time with your work intact. That's not a feature - that's a philosophy.

I wanted that discipline for my own publishing workflow. So I built it into something I could run every day.

What we turned it into

Shareables MCP is a local Model Context Protocol server that turns approved content into validated, staged HTML shareables - and hands them off to GitHub Pages.

Claude and Codex are the operators. The pipeline is the lane.

It went through four versions. Each one added something the previous version couldn't do.

Version by version

V1 - The pipeline

Local MCP server, manifest system, validation, staging handoff to GitHub Pages, Obsidian control area. The skeleton.

V1.1 - The look

A locked visual style: paper background, teal/coral/leaf/gold tokens, card layout. A non-sensitive unlisted demo shipped.

V1.2 - The renderer

Replaced hand-written HTML with structured rendering from a ShareableSpec. The MCP owns the shell; agents send content.

V1.3 - The intake

Source intake, content bundles, image registration, preflight checks, review packets, and workflow wizard tools.

V1.4 - The vault

A dedicated Kyle Shareables Obsidian vault with MCP tools for project records, cut records, shareable sync, review notes, and search.

Version by version visual

How it works now

  1. Intake - source files are ingested into a compact content bundle. Extracted, hashed, marked approved.
  2. Render - structured content (title, sections, cards, steps) is sent to the MCP. It owns the HTML shell, style, and layout.
  3. Validate - private paths, branding errors, and missing decisions are caught before anything leaves the machine.
  4. Review - a review packet is created locally. Jonathan reads it. Feedback and publish decisions are recorded.
  5. Stage - if confirmed public, the MCP places the HTML in the GitHub Pages working tree. A push step publishes it.
  6. Share - unlisted pages get a public-by-link URL with noindex. Listed pages appear on the home index.

What stayed true to your packet

Local-first

Everything runs on local hardware. Nothing leaves the machine until Jonathan explicitly confirms public staging.

Provide an exit

Content stays as markdown. Specs are JSON. The pipeline can be inspected, forked, or walked away from.

Design for sufficiency

One MCP server. One SQLite database. One style system. Nothing optimized for scale it doesn't need.

Reusable lenses

The explainer lens that shaped your packet is the same lens shaping this page. Structure travels.

What changed

MCP tools instead of scripts

Claude and Codex call structured tools. No manual script runs. The pipeline is the conversation.

Structured rendering

Agents send a ShareableSpec - title, sections, cards, steps. The MCP assembles the HTML. No hand-written markup.

Safety gates

Staging requires two explicit confirmations: confirm_public and sensitive_reviewed. The pipeline can't be rushed past them.

Obsidian control surface

A dedicated vault tracks every project, cut, shareable, review, and publish decision. Human-readable, no plugins required.

GitHub Pages handoff

Unlisted URLs with noindex. Listed pages with sitemap. The MCP stages; a separate push step publishes.

A note on this page

This page is unlisted: it won't appear on search engines or the home index, but anyone with the URL can read it. Same tradeoff your packet describes - portable and shareable, not broadcasted.

What this bundle is

This is the portable companion edition of Shareables. It keeps the workflow core: start a shareable, ingest sources, render from structured content, validate, create review packets, and optionally stage for publishing.

It is built for Claude Desktop and asks for your own folders during install. It does not carry over my vault, repo defaults, filesystem paths, or private history.

Install in Claude Desktop

  1. Download shareables-personal-v1.mcpb from this page.
  2. Open Claude Desktop, go to Settings, then Extensions.
  3. Drag and drop the .mcpb file into Claude Desktop or use the Install Extension flow.
  4. Fill in your own workspace root and any optional folders you want: site output, public base URL, Obsidian vault root, and packet references.
  5. Restart or reload Claude Desktop so the Shareables tools appear.

What it includes

Shareable workflow core

Draft manifests, structured rendering, validation, review packets, and optional publish staging.

Optional Obsidian integration

Vault support stays available, but only when the user configures their own vault path.

Optional publish staging

GitHub Pages style staging stays possible when the user provides a site output folder and public base URL.

What it does not include

My vault

It does not install Jonathan's Obsidian vault or write to Jonathan's app registry.

My repo defaults

It does not assume Jonathan's repo names, local paths, or publishing folders.

My private history

It does not carry private conversation history or data from the private full-power MCP.

Build your own instead

If you would rather have Claude build your own version around your folders and style than install my bundle as-is, use the standalone Claude Code prompt.

That prompt starts with an interview in normal Claude chat before any code is written.

What changed in V1.5

Shareables is no longer one implicit house style pretending to fit every page. The renderer now has a small, explicit option system.

That option system is constrained on purpose: three template families and three style sources, with the packet shell used here to keep story, install flow, and technical explanation inside one standalone HTML file.

Template families

Editorial

Long-form narrative shareables with a strong hero and flowing sections.

Docs

Technical explainers, setup guides, and denser reference pages with a sticky section rail.

Packet

One standalone HTML file with top-level pages like Getting Started, Install, and Technical.

Style sources

Shareables Kyle

Paper-and-ink utility shell for fast explainers and clean public pages.

Megan Editorial

Warm cream editorial language with stronger nav and offset-shadow personality.

Bridge

A warmer mixed house style: Megan typography and rhythm, Shareables hierarchy and discipline.
