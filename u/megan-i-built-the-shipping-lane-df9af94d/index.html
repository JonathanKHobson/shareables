The short version

You shared the Local Intelligence Packet: a portable way to turn selected work into curated HTML frames without locking the work into one app or one folder layout.

I turned that idea into Shareables MCP: a local pipeline that manages manifests, rendering, validation, review state, optional Obsidian records, and GitHub Pages staging.

This page was rebuilt with that lane, and now it also carries a portable Shareables extension you can drag into Claude Desktop plus a Claude Code prompt for building your own version from scratch.

What your packet gave me

Cuts

A shaping config: who it is for, what matters, what gets emphasized, and how it should be framed.

Frames

Self-contained HTML outputs that travel cleanly and open anywhere.

Lenses

Reusable structures like explainer, pitch, product website, first look, and timeline.

Windows

Multiple frames or views grouped into a single reader-friendly artifact.

Person-aware publishing

A reminder that the same source material should read differently for different people.

Local-first principles

Portable files, visible tradeoffs, exits built in, and enough system to ship without turning everything into infrastructure.

What I turned it into

The private full-power lane

My local Python Shareables MCP manages manifests, content bundles, image assets, validation, staging, and the dedicated Obsidian control vault.

Structured rendering

Claude and Codex do not hand-style whole pages anymore. They provide structured content and the MCP renders the HTML shell.

Review and safety gates

Public staging requires explicit confirmation and sensitive review. Unlisted stays public-by-link, never private.

Portable companion bundle

A new Node-based MCPB extension strips out my personal defaults and exposes the same lane with user-configured folders, optional Obsidian, and optional publish staging.

How the shipping lane works now

  1. Start a shareable manifest with audience, purpose, visibility, and source pointers.
  2. Ingest source files into a compact content bundle so future agent prompts stay short.
  3. Register approved assets with alt text and stable asset IDs.
  4. Render from structured content instead of hand-authoring full HTML and CSS.
  5. Validate style, private-path leaks, asset issues, and publish gates.
  6. Stage to a public site tree only after an explicit public-safety checkpoint.

Take it with you

The portable Shareables extension attached here is a generalized companion edition. It keeps the workflow, the style system, optional Obsidian sync, and optional publish staging, but it does not carry over my private vault, repo defaults, filesystem paths, or conversation history.

Install in Claude Desktop

  1. Download shareables-personal-v1.mcpb from this page.
  2. Open Claude Desktop, go to Settings, then Extensions.
  3. Drag and drop the .mcpb file into Claude Desktop or use the Install Extension flow.
  4. Fill in your own workspace root and any optional folders you want: site output, public base URL, Obsidian vault root, and packet references.
  5. Restart or reload Claude Desktop so the Shareables tools appear.

Build your own instead

If you do not want my bundle at all, use the attached Claude Code prompt instead. It tells Claude to start with a real interview in chat before writing code.

That prompt explicitly asks about your current folder layout, example files, style references, publishing preferences, and whether you want Obsidian support. Then it makes Claude summarize the intended MCP before it builds anything.

That means you can use the same lane without inheriting my machine or my preferences.

What to show Claude when you build your own

  • Where your source material lives today: one root folder, many folders, or a mixed setup.
  • Two to five example files or directories you want supported first.
  • Style references: sites, screenshots, palettes, or existing shareables you already like.
  • Whether you want local-only, unlisted public-by-link, or full publish staging.
  • Whether you want a dedicated Obsidian vault, a section in an existing vault, or no Obsidian support at all.

What this bundle includes

Shareable workflow core

Manifest creation, structured rendering, validation, review packets, and publish handoff.

Optional Obsidian support

A configurable vault workflow that writes only inside the vault you choose and never touches the app registry.

Optional publish staging

If you configure a site output folder and public base URL, the extension can stage listed and unlisted shareables there.

Bundled prompts

One prompt for building your own Shareables MCP and one prompt for making your first shareable.

What it does not include

My vault

No Kyle Shareables vault content or Jonathan-specific Obsidian defaults.

My repo defaults

No hard-coded /Volumes paths, repo paths, or private site assumptions.

My private history

No private source notes, no local conversation history, and no personal data bundled in the extension.

Automatic privacy guarantees

Unlisted is still public by URL. The review checkpoint still matters.
